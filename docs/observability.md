# Observability (logging, correlation) — STN backend

Версия: **1.0**  
Дата: **2025-12-20**

## 1. Цель и область применения

Документ фиксирует минимальные и обязательные требования по наблюдаемости (observability) backend-сервиса STN:

- единый **Request ID / Correlation ID** для HTTP-запросов;
- единые правила **логирования** (формат, обязательные поля, уровни);
- правила **безопасности логов** (не логировать секреты/PII);
- требования к **тестам**, подтверждающим корректность механизма;
- рекомендации для WebSocket (без изменения публичного WS-контракта).

Этот документ является нормативной основой для реализации механизма из этапа **0.6** (Correlation ID и логирование).

> Примечание: разделы «Метрики» и «Трейсинг» приведены как рекомендации/задел. Их реализация опциональна, если не требуется в текущем этапе.


## 1.1. Принятые решения для проекта (фиксировано)

Следующие решения **зафиксированы** для реализации (изменять только по отдельному решению/миграции документации):

- Формат логов по умолчанию: **text (key=value)**.
- Политика access log в проде: **errors+slow**  
  - логировать запросы со статусом **`>= 400`** и запросы с **`durationMs >= SLOW_THRESHOLD_MS`**;
  - запросы `2xx/3xx` быстрее порога — не логировать на INFO.
- Дублирование `requestId` в теле `ErrorResponse` **не требуется** (достаточно заголовка `X-Request-Id`).

Значение `SLOW_THRESHOLD_MS` по умолчанию: **500 ms** (может быть параметризовано).

---

## 2. Термины

- **Request ID** — идентификатор конкретного HTTP-запроса (синоним: correlationId).
- **MDC** (Mapped Diagnostic Context) — контекст логгера, привязанный к потоку выполнения (thread-local), из которого логгер подставляет значения в каждую строку лога.
- **Access log** — одиночная строка лога на каждый HTTP-запрос (метод/путь/статус/время).

---

## 3. Request ID (HTTP correlation)

### 3.1. Заголовок

- Входящий заголовок: **`X-Request-Id`**
- Исходящий заголовок: **`X-Request-Id`** (всегда должен присутствовать в ответе)

### 3.2. Правила генерации и переиспользования

1) Если клиент прислал `X-Request-Id` и он валиден — сервер использует его.  
2) Если заголовка нет или он невалиден — сервер **генерирует новый** Request ID.  
3) Сервер всегда возвращает фактически используемый Request ID в ответе в `X-Request-Id`.

Рекомендуемый формат генерации: **UUID v4** (строка). Допускаются иные форматы (ULID), но должны удовлетворять правилам валидации ниже.

### 3.3. Валидация входящего `X-Request-Id`

Чтобы избежать «засорения» логов и потенциальных атак на парсинг/хранилище логов:

- максимальная длина: **128** символов;
- допустимые символы: латиница/цифры и `- _ . :`  
  Регулярное выражение (ориентир): `^[A-Za-z0-9._:-]{1,128}$`

Если значение не проходит валидацию — оно **игнорируется**, и Request ID генерируется сервером.

### 3.4. Хранение в MDC

Request ID должен быть помещён в MDC под ключом:

- **`requestId`**

И должен присутствовать в каждой строке лога, относящейся к обработке запроса.

### 3.5. Очистка MDC

После обработки запроса запись `requestId` должна быть **удалена из MDC** в блоке `finally`, чтобы избежать утечек контекста между запросами при использовании пула потоков.

## 4. Логирование

### 4.1. Общие требования

- Логи должны быть **детерминированно пригодны** для расследования инцидентов.
- Все строки лога, относящиеся к обработке HTTP-запроса, должны содержать `requestId`.
- Логи не должны содержать секретов (см. раздел 6).

### 4.2. Формат логов

Допускаются два формата:

1) **Текстовый (key=value)** — минимальные требования; проще в настройке.  
2) **JSON-логирование** — рекомендуется для централизованных систем (ELK/Graylog/Loki).

В текущей реализации допускается текстовый формат при соблюдении обязательных полей (ниже). Переход на JSON не должен менять семантику полей.

### 4.3. Обязательные поля в каждой строке лога

Минимальный набор:

- `timestamp` (ISO 8601, UTC или локально-согласованный формат логгера)
- `level`
- `logger` (или class)
- **`requestId`** (из MDC), если лог относится к HTTP-запросу
- `message`

Рекомендуемые дополнительные поля (если доступны):

- `userId` (после успешной аутентификации)
- `method`, `path`
- `status`, `durationMs` (в access log)
- `remoteIp` (аккуратно, без PII-избыточности)

### 4.4. Access log (лог на каждый запрос)

Сервер должен писать одну строку access log **после** обработки запроса с полями:

- `requestId`
- `method`
- `path`
- `status`
- `durationMs`

Политика по уровню:
- по умолчанию: **INFO**
- допускается настройка (prod):  
  - INFO только для `status >= 400` и/или `durationMs >= SLOW_THRESHOLD_MS`  
  - DEBUG для остальных

Порог «медленного» запроса (по умолчанию): **500 ms**.

### 4.5. Логи ошибок и исключений

- Для необработанных исключений (HTTP 500) необходимо:
  - логировать `requestId`,
  - логировать stacktrace на уровне **ERROR**,
  - в ответ клиенту возвращать `ErrorResponse` согласно `errors-spec.md` (Request ID остаётся в заголовке `X-Request-Id`).

- Для ожидаемых бизнес-ошибок (400/401/403/404/409):
  - допускается лог на уровне **WARN** или отсутствие лога (если access log уже фиксирует статус и requestId).
  - запрет на логирование чувствительных данных из входных DTO.

## 5. Observability для WebSocket (без изменения публичного контракта)

Публичный WS-контракт не расширяется ради observability. Требования ограничиваются серверным логированием.

### 5.1. Идентификаторы

Для логов WebSocket сервер должен фиксировать:

- `wsSessionId` — ID сессии (генерируется сервером или берётся из session id фреймворка)
- `userId` — аутентифицированный пользователь
- `channel` / `channelId` — если применимо
- `eventType` — для отправляемых событий

Если клиентское сообщение содержит какой-либо идентификатор (например, `clientMessageId`) — логировать его как дополнительное поле, но **не требовать** его наличия.

### 5.2. Привязка к Request ID

Для WS handshake можно:
- создать отдельный `requestId` на момент handshake и логировать его как `handshakeRequestId`,
- но не требуется переносить его на каждое сообщение.

---

## 6. Безопасность логов (запреты)

Строго запрещено логировать:

- пароли в любом виде;
- JWT/refresh tokens и любые секреты;
- полные значения заголовка `Authorization`;
- приватные ключи, строки подключения, креды Redis/Postgres;
- содержимое полей, которые могут содержать PII/секреты, без маскирования.

Рекомендация по PII:
- email/никнейм допускаются только если реально нужны для диагностики; предпочтительно логировать `userId`.
- если нужно логировать email — маскировать (например, `u***@domain.com`).

---

## 7. Пропагация Request ID во внешние вызовы (рекомендация)

Если backend делает исходящие HTTP-вызовы (в будущем):

- во все исходящие запросы добавлять заголовок `X-Request-Id` со значением текущего `requestId` из MDC;
- при логировании исходящих вызовов включать `requestId`, `targetHost`, `path`, `status`, `durationMs`.

---

## 8. Асинхронность и MDC (важно при @Async)

MDC является thread-local. Если используется `@Async`, `CompletableFuture`, планировщики — `requestId` **не перенесётся автоматически**.

Требование:
- при использовании асинхронных исполнителей необходимо внедрить механизм переноса MDC (например, `TaskDecorator`) и тест/проверку на отсутствие утечек.

Если асинхронность не используется — реализация не требуется.

---

## 9. Требования к тестам

Минимальные тесты для HTTP:

1) **Server-generated Request ID**  
   - запрос без `X-Request-Id` возвращает ответ с `X-Request-Id` (не пустой).

2) **Echo Request ID**  
   - запрос с `X-Request-Id=abc-123` возвращает тот же `abc-123`.

3) **Invalid Request ID**  
   - запрос с невалидным `X-Request-Id` (например, слишком длинным или с пробелом) приводит к генерации нового значения.

4) **No MDC leakage**  
   - два последовательных запроса должны иметь разные `requestId` (если клиент не задавал свой), и значения не должны «перетекать».

## 10. Минимальная реализация в Spring Boot (ориентир)

### 10.1. HTTP Filter

Использовать `OncePerRequestFilter`:

- до `filterChain.doFilter(...)`:
  - определить requestId (header или генерация),
  - `MDC.put("requestId", requestId)`,
  - `response.setHeader("X-Request-Id", requestId)`,
  - зафиксировать `startTime`.

- после `doFilter(...)`:
  - вычислить `durationMs`,
  - записать access log.

- в `finally`:
  - `MDC.remove("requestId")`

### 10.2. Логгер-конфигурация

В шаблон логов добавить MDC-поле:

- Logback pattern: `%X{requestId}`

Пример (ориентир, не «единственно верный»):

```text
%d{yyyy-MM-dd'T'HH:mm:ss.SSSXXX} %-5level requestId=%X{requestId} [%thread] %logger{36} - %msg%n
```

## 11. Параметры, допускающие настройку (не изменяя семантику)

- `SLOW_THRESHOLD_MS` (по умолчанию 500)
- стратегия access log:
  - always / errors-only / slow-only / sampled
- формат логов:
  - text / json
- допустимый max length для `X-Request-Id` (по умолчанию 128)

---

## 12. Изменение параметров и эволюция

- Переход на JSON-логи допускается в будущем без изменения семантики полей (должны сохраняться `requestId`, `method`, `path`, `status`, `durationMs` и т.д.).
- Изменение политики access log (например, добавление sampling или логирование всех запросов) должно быть оформлено как обновление этого документа.
- `SLOW_THRESHOLD_MS` допускается менять конфигурацией по окружениям (dev/test/prod) при сохранении общей политики **errors+slow**.
