# STN — Спецификация ошибок backend (`ErrorResponse` / `ErrorCode`)

## 1. Назначение документа

Этот документ описывает:
- единый формат ответа об ошибке для REST-API backend-сервиса STN;
- справочник кодов ошибок `ErrorCode`;
- соответствие типовых ситуаций HTTP-статусам и кодам ошибок;
- общие принципы обработки и логирования ошибок.

Документ отражает реализацию этапа 1.1 (введение глобального обработчика ошибок и базовой модели `ErrorResponse`).

## 2. Базовый формат ответа об ошибке

Все REST-эндпоинты STN при ошибках, которые обрабатываются backend’ом (4xx/5xx), возвращают JSON в формате:

```json
{
  "code": "BAD_REQUEST",
  "message": "Validation failed",
  "details": {
    "fieldErrors": [
      { "field": "name", "message": "must not be blank" }
    ]
  },
  "timestamp": "2025-11-28T11:39:00Z"
}
```

### 2.1. Поля `ErrorResponse`

- `code` (`string`, обязательно)  
  Машиночитаемый код ошибки. Значение совпадает с `ErrorCode.name()` на backend’е.  
  Стабильный идентификатор для фронтенда и логирования.

- `message` (`string`, обязательно)  
  Краткое человекочитаемое описание причины ошибки.  
  Не содержит внутренних деталей (stack trace, SQL, пути к файлам и т.п.). Текст может меняться без нарушения контракта.

- `details` (`object | null`, опционально)  
  Дополнительные структурированные данные по ошибке:
  - валидационные ошибки;
  - дополнительные атрибуты контекста (например, идентификаторы сущностей);
  - может быть `null`, если дополнительная информация не требуется.

  Структура `details` зависит от типа ошибки и описана ниже для типичных случаев.

- `timestamp` (`string`, обязательно)  
  Время формирования ответа в формате ISO 8601, UTC, пример: `2025-11-28T11:39:00Z`.

## 3. Справочник `ErrorCode`

Ниже перечислены базовые коды ошибок, используемые на этапе 1.1.  
При необходимости список может быть расширен, но уже существующие коды должны оставаться стабильными.

| Код              | HTTP по умолчанию | Назначение                                                                |
|------------------|-------------------|---------------------------------------------------------------------------|
| `BAD_REQUEST`    | 400               | Некорректный запрос клиента: формат, синтаксис, нарушена валидация и т.п. |
| `UNAUTHORIZED`   | 401               | Требуется аутентификация или неверный/просроченный токен.                 |
| `FORBIDDEN`      | 403               | У пользователя нет прав для выполнения операции.                          |
| `NOT_FOUND`      | 404               | Ресурс не найден.                                                         |
| `CONFLICT`       | 409               | Конфликт состояния (например, нарушение бизнес-ограничения).              |
| `INTERNAL_ERROR` | 500               | Необработанная внутренняя ошибка сервера.                                 |
| `ACCOUNT_BLOCKED`| 403               | Аккаунт заблокирован.                                                     |
| `ACCOUNT_DELETED`| 403               | Аккаунт удален.                                                           |

В дальнейшем могут быть добавлены специализированные коды (например, `VALIDATION_ERROR`, `METHOD_NOT_ALLOWED`), но они должны оставаться «техническими» и не привязанными к конкретному домену (отряды, роты и т.п.).

## 4. Соответствие типовых ситуаций HTTP-статусам и ErrorCode

Глобальный обработчик ошибок (этап 1.1) маппит типовые исключения Spring/Java в HTTP-ответы.

### 4.1. Валидационные ошибки запроса

Классы:
- `MethodArgumentNotValidException`
- `BindException`
- `ConstraintViolationException`

Ответ:

- HTTP: `400 Bad Request`
- `code`: `BAD_REQUEST` (или `VALIDATION_ERROR`, если он будет введён)
- `message`: обобщённое сообщение, например `Validation failed`
- `details`:

```json
{
  "fieldErrors": [
    { "field": "name", "message": "must not be blank" },
    { "field": "lat", "message": "must be between -90 and 90" }
  ]
}
```

Рекомендуемый формат:

```ts
details = {
  fieldErrors: Array<{
    field: string;    // имя поля (JSON / DTO)
    message: string;  // локализованное или англоязычное сообщение
  }>
}
```

### 4.2. Неверный формат тела запроса (JSON)

Класс:
- `HttpMessageNotReadableException`

Примеры:
- битый JSON;
- несовпадение типов (`"abc"` вместо числа).

Ответ:

- HTTP: `400 Bad Request`
- `code`: `BAD_REQUEST`
- `message`: что-то вроде `Malformed JSON request`
- `details`: по умолчанию `null` (при необходимости можно добавить более детальную информацию, но без утечек внутренних деталей парсера).

### 4.3. Неподдерживаемый HTTP-метод

Класс:
- `HttpRequestMethodNotSupportedException`

Ответ:

- HTTP: `405 Method Not Allowed`
- `code`: либо `BAD_REQUEST`, либо специализированный `METHOD_NOT_ALLOWED` (если введён).
- `message`: `HTTP method not allowed for this endpoint`.
- `details`: может содержать список допустимых методов:

```json
{
  "allowedMethods": ["GET", "POST"]
}
```

### 4.4. Не найден endpoint (404)

Класс (если включена соответствующая конфигурация Spring):
- `NoHandlerFoundException`

Ответ:

- HTTP: `404 Not Found`
- `code`: `NOT_FOUND`
- `message`: `Resource not found`
- `details`: опционально URI/метод запроса (без внутренних технических деталей).

### 4.5. Необработанные ошибки (500)

Класс:
- любой `Exception`, не пойманный более специализированными хэндлерами.

Ответ:

- HTTP: `500 Internal Server Error`
- `code`: `INTERNAL_ERROR`
- `message`: обобщённое сообщение, например `Internal server error`
- `details`: `null` или минимальный контекст, который не раскрывает внутреннюю структуру приложения.

Все такие ошибки должны логироваться на сервере с уровнем `ERROR` и полным stack trace.

## 5. Ошибки аутентификации и авторизации

Ошибки безопасности (JWT, доступ к защищённым ресурсам) обрабатываются слоем Spring Security и/или кастомным фильтром/хэндлером.

Рекомендуемые соответствия:

- Неаутентифицированный запрос к защищённому ресурсу:
  - HTTP: `401 Unauthorized`
  - `code`: `UNAUTHORIZED`
  - `message`: `Authentication required` или аналогичное.

- Аутентифицированный пользователь без нужных прав:
  - HTTP: `403 Forbidden`
  - `code`: `FORBIDDEN`
  - `message`: `Access denied` или аналогичное.

Структура `ErrorResponse` остаётся той же.

## 6. Доменно-специфичные ошибки

На текущем этапе (1.1) доменно-специфичные коды (`SQUAD_NOT_FOUND`, `COMPANY_CLOSED`, и т.п.) **не вводятся**.  
Вся доменная логика маппится на базовые коды:

- `NOT_FOUND` — для отсутствующих доменных сущностей;
- `CONFLICT` — для нарушений доменных ограничений (нельзя вступить в два отряда, нельзя передать командование самому себе и т.д.);
- `FORBIDDEN` — для нарушений прав (операция запрещена текущей роли).

Если в дальнейшем возникнет потребность в стабильных доменных кодах (для сложной клиентской обработки), это будет оформлено отдельным расширением `ErrorCode` и обновлением данного документа.

## 7. Общие принципы

1. **Стабильность кода ошибки (`code`)**  
   Поле `code` считается частью публичного контракта и должно меняться только при явном мигрировании клиентов.

2. **Сообщение (`message`) не является контрактом**  
   Поле `message` предназначено для отображения пользователю или логирования, текст не фиксируется жёстко.

3. **Логирование**  
   - Валидационные и ожидаемые ошибки допускается логировать уровнем `WARN` или не логировать.
   - Неожиданные ошибки (`INTERNAL_ERROR`) всегда логируются с уровнем `ERROR` и stack trace.

4. **Отсутствие внутренних деталей**  
   В ответах нельзя возвращать SQL-запросы, части stack trace, пути файлов, внутренние идентификаторы сервиса и т.п.

5. **Локализация**  
   На уровне контракта не фиксируется язык `message`. В прототипе допускается английский.  
   Если локализация потребуется, будет описано отдельно (например, `code` остаётся стабильным, локализация происходит на клиенте).
