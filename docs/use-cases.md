
# Use Cases: Тактический навигационный и коммуникационный сервис

Этот документ описывает ключевые пользовательские сценарии (use cases) для приложения тактической навигации и взаимодействия, основанные на утверждённом техническом задании.

## Акторы

- **Гость** — неавторизованный пользователь.
- **Пользователь (User)** — авторизованный пользователь с ролью `USER`, `MODERATOR` или `ADMIN`.
- **Солдат (Soldier)** — пользователь, состоящий в отряде, не являющийся его командиром.
- **Командир отряда (Commander)** — пользователь, являющийся командиром конкретного отряда.
- **Отряд (Squad)** — логический участник в сценариях, когда речь идёт о группе пользователей.
- **Рота (Company)** — логический участник, когда речь идёт об объединении отрядов.

---

## UC-1. Регистрация и вход в систему

**Акторы:** Гость, Пользователь.

**Предусловия:**

- Гость не авторизован в системе.

**Основной сценарий:**

1. Гость открывает приложение и выбирает действие «Регистрация».
2. Гость вводит логин/email, пароль, позывной (`nickname`).
3. Система:
   - проверяет уникальность логина/email;
   - создаёт сущность `User` с:
     - `systemRole = USER`,
     - случайным `avatarIcon` из предопределённого набора,
     - `status = ALIVE`;
   - сохраняет пользователя в БД.
4. Пользователь переходит к экрану входа и вводит логин/email и пароль.
5. Система:
   - проверяет корректность логина/email и пароля;
   - при успехе генерирует JWT-токен;
   - возвращает JWT клиенту.
6. Клиент сохраняет токен и использует его для последующих защищённых запросов.

**Альтернативы/ошибки:**

- Введённый логин/email уже зарегистрирован:
  - Система отклоняет регистрацию, возвращает ошибку «учётная запись уже существует».
- Неверный логин/email или пароль при входе:
  - Система отклоняет запрос входа, возвращает ошибку авторизации.

## UC-2. Создание отряда

**Акторы:** Пользователь.

**Предусловия:**

- Пользователь авторизован (имеет валидный JWT).
- Пользователь не состоит ни в одном отряде.

**Основной сценарий:**

1. Пользователь выбирает действие «Создать отряд».
2. Пользователь при необходимости вводит описание отряда (`description`).
3. Система:
   - проверяет, что пользователь не состоит в отряде;
   - генерирует имя отряда в формате `Отряд [хищное животное или ядовитая змея]`;
   - выбирает цвет отряда `color` из палитры («радуга + белый»);
   - создаёт сущность `Squad`:
     - `commander = пользователь`,
     - `members = { пользователь }`,
     - `isOpen = true`,
     - `company = null`;
   - связывает пользователя с этим отрядом как с текущим;
   - сохраняет отряд в БД.
4. Система отправляет пользователю уведомление о создании отряда и вступлении в него (`SQUAD_CREATED`, `JOINED_SQUAD`).

**Альтернативы/ошибки:**

- Пользователь уже состоит в отряде:
  - Система отклоняет попытку создания отряда и возвращает соответствующую ошибку.

## UC-3. Вступление в отряд

**Акторы:** Пользователь, Отряд.

**Предусловия:**

- Пользователь авторизован.
- Пользователь не состоит ни в одном отряде.
- В системе существует хотя бы один отряд с `isOpen = true`.

**Основной сценарий:**

1. Пользователь запрашивает список доступных открытых отрядов.
2. Система возвращает список отрядов с `isOpen = true`.
3. Пользователь выбирает отряд для вступления.
4. Клиент отправляет запрос «Вступить в отряд {squadId}».
5. Система:
   - проверяет, что пользователь не состоит в отряде;
   - проверяет, что указанный отряд существует и имеет `isOpen = true`;
   - добавляет пользователя в `Squad.members`;
   - связывает пользователя с этим отрядом как с текущим;
   - сохраняет изменения.
6. Система отправляет пользователю уведомление `SQUAD_MEMBER_JOINED`. При необходимости может отправить уведомления другим участникам отряда.

**Альтернативы/ошибки:**

- Пользователь уже состоит в отряде:
  - Система отклоняет вступление и возвращает ошибку.
- Отряд стал закрытым (`isOpen = false`) после получения списка, но до попытки вступления:
  - Система отклоняет вступление и возвращает ошибку.
- Отряд был распущен:
  - Система возвращает ошибку «отряд не найден».

## UC-4. Выход из отряда и передача командования

**Акторы:** Солдат, Командир отряда.

**Предусловия:**

- Пользователь авторизован.
- Пользователь состоит в отряде (как солдат или командир).

### Основной сценарий: выход солдата

1. Солдат открывает экран отряда и выбирает действие «Покинуть отряд».
2. Клиент отправляет запрос на выход из текущего отряда.
3. Система:
   - удаляет пользователя из `Squad.members`;
   - если после удаления множество `members` пусто:
     - удаляет отряд;
     - каскадно удаляет связанные сущности (например, приказы отряда);
     - формирует событие `SQUAD_DISBANDED`.
4. Система отвязывает пользователя от отряда (у пользователя больше нет текущего отряда).
5. Пользователь получает уведомление `SQUAD_MEMBER_LEFT`.

### Основной сценарий: выход командира

1. Командир открывает экран отряда и выбирает:
   - либо «Распустить отряд»,
   - либо «Покинуть отряд».
2. Если выбран вариант «Распустить отряд»:
   - система удаляет `Squad` и все связанные приказы;
   - все участники отряда получают уведомление `SQUAD_DISBANDED`;
   - у всех участников пропадает связь с отрядом.
3. Если выбран вариант «Покинуть отряд»:
   - система проверяет наличие других участников в `members` (кроме командира):
     - если есть другие участники:
       - выбирает одного участника (например, первого по id или по списку);
       - назначает этого участника `commander`;
       - удаляет текущего командира из `members`;
       - отвязывает его от отряда;
       - новому командиру отправляется уведомление `BECAME_COMMANDER`;
       - старому командиру — `LEFT_SQUAD`;
     - если других участников нет:
       - отряд удаляется полностью, как в сценарии роспуска.

**Альтернативы/ошибки:**

- Пользователь не состоит ни в одном отряде:
  - Система отклоняет запрос и возвращает ошибку.

## UC-5. Создание роты и вступление отрядом в роту

**Акторы:** Командир отряда, Отряд, Рота.

**Предусловия:**

- Пользователь авторизован.
- Пользователь является командиром отряда.
- Отряд не состоит в роте (для создания новой роты или вступления в существующую).

### Основной сценарий: создание роты

1. Командир отряда открывает экран управления и выбирает «Создать роту».
2. Командир указывает описание роты и флаг открытости (`isOpen`).
3. Система:
   - генерирует имя роты в формате `Рота [крупная река или водоём в РФ]`;
   - создаёт сущность `Company`:
     - `name`,
     - `description`,
     - `isOpen`,
     - `squads = { отряд-инициатор }`;
   - устанавливает связь отряда с ротой (`Squad.company = Company`).
4. Система сохраняет изменения и может отправить участникам отряда уведомление о вступлении в роту (`SQUAD_JOINED_COMPANY`).

### Основной сценарий: вступление отрядом в существующую роту

1. Командир отряда запрашивает список доступных рот с `isOpen = true`.
2. Система возвращает список открытых рот.
3. Командир выбирает роту для вступления.
4. Клиент отправляет запрос «Вступить отрядом в роту {companyId}».
5. Система:
   - проверяет, что отряд не состоит в другой роте;
   - проверяет, что рота существует и `isOpen = true`;
   - добавляет отряд в `Company.squads`;
   - устанавливает `Squad.company = Company`;
   - сохраняет изменения.
6. Система отправляет участникам отряда уведомление `SQUAD_JOINED_COMPANY`.

**Альтернативы/ошибки:**

- Отряд уже состоит в роте:
  - Система отклоняет запрос.
- Рота закрыта (`isOpen = false`) или удалена:
  - Система возвращает соответствующую ошибку.

## UC-6. Постановка метки ENEMY_SPOTTED

**Акторы:** Солдат, Командир отряда.

**Предусловия:**

- Пользователь авторизован.
- Пользователь состоит в отряде.
- Тип метки `ENEMY_SPOTTED` активен в системе (`active = true`).
- У пользователя есть выбранная позиция на карте (по автоматическим координатам или вручную).

**Основной сценарий:**

1. Пользователь открывает карту и выбирает действие «Обнаружен противник».
2. Клиент определяет координаты для метки:
   - либо по текущей автоматической геопозиции пользователя,
   - либо по точке, выбранной кликом на карте.
3. Клиент отправляет запрос на создание тактической метки типа `ENEMY_SPOTTED` с координатами и (опционально) пользовательским описанием.
4. Система:
   - проверяет, что пользователь состоит в отряде;
   - проверяет, что тип `ENEMY_SPOTTED` доступен для `ANY_MEMBER`;
   - создаёт сущность `TacticalMarker`:
     - `markerType = ENEMY_SPOTTED`,
     - `author = пользователь`,
     - `squad = текущий отряд пользователя`,
     - `company = рота отряда (если отряд состоит в роте и пользователь при создании метки поставил флаг "транслировать в роту")`,
     - `lat/lng` — координаты;
     - `description` — либо дефолтное, либо заданное пользователем;
     - `createdAt = now`,
     - `expiresAt = now + 10 минут`.
   - сохраняет метку.
5. Через WebSocket/уведомления:
   - метка становится видна:
     - всем членам отряда пользователя,
     - всем членам других отрядов той же роты (если отряд состоит в роте).

**Альтернативы/ошибки:**

- Пользователь не состоит в отряде — запрос отклоняется.
- Тип `ENEMY_SPOTTED` деактивирован (`active = false`) — запрос отклоняется.

## UC-7. Создание приказа и отметка его выполнения

**Акторы:** Командир отряда, Солдаты.

**Предусловия:**

- Пользователь авторизован.
- Пользователь является командиром отряда.

### Основной сценарий: создание приказа

1. Командир открывает экран приказов своего отряда и выбирает «Создать приказ».
2. Командир вводит текст приказа (`description`).
3. Клиент отправляет запрос на создание приказа.
4. Система:
   - проверяет, что пользователь — командир отряда;
   - создаёт сущность `Order`:
     - `squad = текущий отряд`,
     - `author = командир`,
     - `description`,
     - `status = ACTIVE`,
     - `createdAt = now`,
     - `completedAt = null`;
   - сохраняет приказ в БД;
   - отправляет событие `ORDER_CREATED` в канал отряда.
5. У всех членов отряда приказ появляется в списке приказов со статусом «активен».

### Основной сценарий: отметка приказа как выполненного

1. Командир выбирает приказ в списке приказов отряда.
2. Командир инициирует действие «Отметить как выполнен».
3. Клиент отправляет запрос на изменение статуса приказа.
4. Система:
   - проверяет, что инициатор — командир отряда (или ADMIN/MODERATOR);
   - проверяет, что текущий статус приказа `ACTIVE`;
   - устанавливает:
     - `status = COMPLETED`,
     - `completedAt = now`;
   - сохраняет изменения;
   - отправляет событие `ORDER_STATUS_CHANGED` в канал отряда.
5. У всех членов отряда приказ отображается со статусом «выполнен».

**Альтернативы/ошибки:**

- Попытка создать приказ пользователем, не являющимся командиром отряда:
  - Система отклоняет запрос.
- Попытка изменить статус приказа пользователем, не являющимся командиром или администратором/модератором:
  - Система отклоняет запрос.
- Попытка изменить статус уже `COMPLETED`:
  - Система возвращает ошибку или игнорирует повторное действие (идемпотентность).

## UC-8. Обновление геопозиции игрока

**Акторы:** Пользователь.

**Предусловия:**

- Пользователь авторизован.
- У пользователя выбран режим работы геолокации (автоматический или ручной).

### Основной сценарий: автоматический режим

1. Клиент запрашивает у системы/браузера доступ к геолокации устройства.
2. Пользователь даёт разрешение на доступ к геоданным.
3. Клиент через заданные интервалы времени получает текущие координаты (`lat/lng`) от устройства.
4. Клиент отправляет на backend запрос с текущими координатами.
5. Система:
   - создаёт новую запись `GeoLocation` для каждого запроса для пользователя с последними координатами и временем;
   - может формировать события/обновления для клиентов отряда (через WebSocket).

### Основной сценарий: ручной режим

1. Пользователь переключается в ручной режим и открывает карту.
2. Пользователь кликом по карте или перемещением маркера указывает своё положение.
3. Клиент отправляет на backend координаты выбранной точки.
4. Система:
   - обновляет запись `GeoLocation` для пользователя;
   - может формировать события/обновления для клиентов отряда (через WebSocket).

**Использование геопозиции:**

- При запросе данных для карты или при получении соответствующих событий по WebSocket:
  - члены отряда видят актуальные позиции друг друга;
  - командиры отрядов, входящих в одну роту, отображаются для всех членов роты.

## UC-9. Изменение статуса ALIVE/DEAD и аватара пользователем

**Акторы:**  
Пользователь (User).

**Предусловия:**

- Пользователь авторизован.

### Основной сценарий: изменение статуса ALIVE/DEAD

1. Пользователь открывает экран профиля или настроек.
2. Пользователь выбирает новый статус: «жив» (`ALIVE`) или «мертвяк» (`DEAD`).
3. Клиент отправляет на backend запрос с желаемым статусом.
4. Система:
   - аутентифицирует пользователя;
   - проверяет, что пользователь изменяет только **свой** статус;
   - обновляет поле `status` в сущности `User`;
   - при необходимости публикует событие об обновлении статуса в отрядный/ротный канал (для обновления UI других игроков).
5. Клиенты других участников отряда получают событие и обновляют отображение пользователя (иконка, цвет, пометка «мертвяк» и т.п.).

**Альтернативы/ошибки:**

- Попытка изменить статус другого пользователя → система отклоняет запрос (ошибка авторизации/доступа).
- Невалидное значение статуса → ошибка валидации.

### Основной сценарий: изменение аватара

1. Пользователь открывает экран выбора аватара.
2. Клиент при необходимости запрашивает у backend список доступных аватар-иконок (если набор иконок не зашит на клиенте).
3. Пользователь выбирает новый аватар из доступного списка.
4. Клиент отправляет на backend запрос с выбранным `avatarIcon`.
5. Система:
   - аутентифицирует пользователя;
   - проверяет, что указанный `avatarIcon` входит в допустимый список;
   - сохраняет новое значение `avatarIcon` в сущности `User`.
6. На карте, в чатах и списках участников у данного пользователя отображается новый аватар.

**Альтернативы/ошибки:**

- Попытка указать несуществующий или запрещённый аватар → система возвращает ошибку валидации.
- Попытка изменить аватар другого пользователя → ошибка доступа.

## UC-10. Управление параметрами отряда и кик участника командиром

**Акторы:**  
Командир отряда (Commander), Модератор/Администратор (MODERATOR/ADMIN), Солдаты (Soldier).

**Предусловия:**

- Пользователь авторизован.
- Для действий командира: пользователь является командиром конкретного отряда.
- Для действий модератора/админа: пользователь имеет соответствующую роль и права модерации отрядов.

### Основной сценарий: изменение параметров отряда командиром

1. Командир открывает экран настроек своего отряда.
2. Командир изменяет один или несколько параметров:
   - `name` — пользовательское имя отряда вместо сгенерированного по умолчанию;
   - `description` — описание отряда;
   - `isOpen` — флаг открытости отряда для вступления.
3. Клиент отправляет запрос на backend с изменёнными параметрами.
4. Система:
   - аутентифицирует пользователя;
   - проверяет, что пользователь является командиром данного отряда (или MODERATOR/ADMIN с правами на этот отряд);
   - валидирует новые значения (длина имени, описание и т.п.);
   - обновляет сущность `Squad` в БД;
   - при необходимости публикует событие в отрядный/ротный каналы (изменение имени/открытости).
5. Все участники отряда (и отряды роты, если нужно) при следующем запросе или через realtime-события видят обновлённые параметры отряда.

**Альтернативы/ошибки:**

- Пользователь не командир отряда и не модератор/админ → система отклоняет запрос.
- Попытка изменить несуществующий отряд → ошибка «отряд не найден».

### Основной сценарий: кик участника из отряда командиром

1. Командир открывает список участников своего отряда.
2. Командир выбирает участника (солдата), которого нужно удалить из отряда, и инициирует действие «Удалить из отряда».
3. Клиент отправляет запрос на backend с идентификатором пользователя, которого нужно исключить.
4. Система:
   - аутентифицирует инициатора;
   - проверяет, что инициатор — командир данного отряда (или MODERATOR/ADMIN с правами модерации);
   - проверяет, что целевой пользователь действительно состоит в этом отряде;
   - удаляет пользователя из `Squad.members` и отвязывает его от отряда;
   - если после удаления в отряде не осталось ни одного участника (теоретический случай) — отряд удаляется полностью.
5. Исключённый пользователь при следующем обращении (или по realtime-событию) видит, что больше не состоит в отряде и должен выбрать новый отряд или создать свой.

**Альтернативы/ошибки:**

- Попытка кикнуть самого себя командиром через этот сценарий → запрос отклоняется (для выхода командир должен использовать сценарий выхода/роспуска).
- Попытка кикнуть пользователя, не состоящего в отряде → ошибка валидации.
- Попытка кикнуть участника другим солдатом без прав → ошибка доступа.

## UC-11. Управление параметрами роты и выход отряда из роты

**Акторы:**  
Командир отряда (Commander), Модератор/Администратор (MODERATOR/ADMIN).

**Предусловия:**

- Пользователь авторизован.
- Отряд пользователя состоит в роте.
- Пользователь является командиром отряда в этой роте (или MODERATOR/ADMIN с правами управления ротами).

### Основной сценарий: изменение параметров роты

1. Командир открывает экран роты, в которой состоит его отряд.
2. Командир изменяет один или несколько параметров роты:
   - `name` — имя роты (по умолчанию сгенерированное, но может быть переопределено);
   - `description` — описание роты;
   - `isOpen` — открыта ли рота для вступления новых отрядов.
3. Клиент отправляет запрос на backend с изменёнными параметрами.
4. Система:
   - аутентифицирует инициатора;
   - проверяет, что он является командиром одного из отрядов роты (или MODERATOR/ADMIN);
   - валидирует новые значения;
   - обновляет сущность `Company`;
   - публикует событие для всех отрядов роты (обновление информации о роте).
5. Все отряды роты при следующем обращении или через события видят обновлённые параметры роты.

**Альтернативы/ошибки:**

- Попытка изменить роту пользователем, чьи отряды не состоят в этой роте → ошибка доступа.
- Попытка изменить удалённую/несуществующую роту → ошибка «рота не найдена».

### Основной сценарий: выход отряда из роты

1. Командир отряда открывает экран роты и выбирает действие «Выйти из роты».
2. Клиент отправляет запрос на backend.
3. Система:
   - аутентифицирует инициатора;
   - проверяет, что он является командиром отряда, состоящего в данной роте;
   - удаляет отряд из списка `Company.squads`;
   - устанавливает `company = null` для отряда;
   - проверяет, остались ли в роте другие отряды:
     - если остались — рота продолжает существовать;
     - если не осталось ни одного отряда — рота удаляется (каскадно, без сохранения «пустой» роты).
4. Участники отряда больше не видят чат и тактические данные на уровне роты, но продолжают работать в рамках своего отряда.

**Альтернативы/ошибки:**

- Попытка вывода отряда пользователем, не являющимся командиром → ошибка доступа.
- Попытка вывести отряд, который не состоит ни в одной роте → ошибка валидации.

## UC-12. Постановка командирских тактических меток (ATTACK/MOVE/HOLD/…)

**Акторы:**  
Командир отряда (Commander).

**Предусловия:**

- Пользователь авторизован.
- Пользователь является командиром отряда.
- Типы командирских меток (`ATTACK`, `MOVE`, `HOLD`, `MISSION_TARGET`, `RESPAWN_POINT`/«Мертвяк» и т.п.) описаны в `TacticalMarkerType`:
  - `roleRestriction = COMMANDER_ONLY`,
  - `canSendToCompany = true` (для меток, которые можно транслировать на уровень роты),
  - `uniquenessPolicy = ONE_PER_USER` или `ONE_PER_SQUAD` для нужных типов.

### Основной сценарий: постановка командирской метки для отряда

1. Командир открывает карту и выбирает тип метки (например, «Атаковать», «Движемся сюда», «Держать позицию»).
2. Командир указывает точку на карте (клик/тап по карте) — координаты `lat/lng`.
3. (Опционально) командир редактирует описание метки (по умолчанию используется текст из `TacticalMarkerType.defaultDescription`).
4. Командир **не** устанавливает флаг отправки в роту (метка только для отряда).
5. Клиент отправляет запрос на backend с:
   - `markerTypeKey` (например, `ATTACK`, `MOVE`, `HOLD`),
   - `lat`, `lng`,
   - `description` (опционально),
   - `sendToCompany = false` или опущен.
6. Система:
   - находит соответствующий `TacticalMarkerType` и проверяет `active = true`;
   - проверяет, что пользователь — командир (`roleRestriction = COMMANDER_ONLY`);
   - применяет `uniquenessPolicy`:
     - если `ONE_PER_USER` — находит ранее созданные этим командиром активные метки данного типа и удаляет/деактивирует их;
     - если `ONE_PER_SQUAD` — находит активные метки данного типа для отряда и удаляет/деактивирует их;
   - создаёт новый `TacticalMarker` с `squad = текущий отряд`, `company = null`, `author = командир`;
   - устанавливает `createdAt = now` и, при наличии `defaultLifetimeSeconds`, `expiresAt = now + defaultLifetimeSeconds`.
7. Все участники отряда получают через WebSocket/уведомления событие о новой метке и видят её на карте.

### Основной сценарий: постановка командирской метки с трансляцией на роту

1. Командир выполняет шаги 1–3, как выше.
2. Командир устанавливает флаг «Отправить в роту».
3. Клиент отправляет запрос на backend с `sendToCompany = true`.
4. Система:
   - дополнительно проверяет, что отряд состоит в роте;
   - если `canSendToCompany = true` для данного `TacticalMarkerType`:
     - устанавливает `company = рота отряда` у создаваемой метки;
   - в остальном логика создания и `uniquenessPolicy` совпадает с предыдущим сценарием.
5. Метка отображается:
   - всем членам отряда (как обычная отрядная метка);
   - всем участникам других отрядов этой роты (как ротная метка).

**Альтернативы/ошибки:**

- Попытка создать командирскую метку пользователем, не являющимся командиром → ошибка доступа.
- Указанный `markerTypeKey` не существует или неактивен → ошибка «тип метки не найден».
- Установлен `sendToCompany = true`, но отряд не состоит в роте → система игнорирует флаг (метка остаётся только отрядной) или возвращает ошибку (по принятой политике).
- Нарушение `uniquenessPolicy` обрабатывается автоматически: старые метки удаляются/деактивируются без отдельного вмешательства пользователя.

## UC-13. Общение в чатах (GLOBAL / SQUAD / COMPANY)

**Акторы:**  
Пользователь (User), Солдат (Soldier), Командир (Commander).

**Предусловия:**

- Пользователь авторизован.
- Для участия в отрядном чате: пользователь состоит в отряде.
- Для участия в ротном чате: отряд пользователя состоит в роте.

### Основной сценарий: подключение к чатам

1. Клиент после авторизации устанавливает WebSocket‑соединение с backend по адресу, например, `/ws/chat`, передавая JWT (в query-параметре или заголовке).
2. Система:
   - аутентифицирует пользователя по токену;
   - привязывает соединение к `userId`;
   - определяет доступные пользователю каналы:
     - всегда: `GLOBAL`;
     - если пользователь состоит в отряде — `SQUAD` для этого отряда;
     - если отряд состоит в роте — `COMPANY` для этой роты;
   - подписывает соединение на соответствующие внутренние каналы (через Redis Pub/Sub или аналоги).
3. Клиент получает от сервера сервисное сообщение/событие (необязательно), что подключение успешно и какие каналы доступны.

### Основной сценарий: отправка сообщения в чат

1. Пользователь выбирает чат в интерфейсе: общий (GLOBAL), отрядный (SQUAD) или ротный (COMPANY).
2. Пользователь вводит текст сообщения.
3. Клиент формирует JSON‑сообщение вида:

   ```json
   {
     "action": "SEND_MESSAGE",
     "channelType": "SQUAD",
     "text": "Двигаемся к точке сбора"
   }
   ```

4. Система:
   - проверяет аутентификацию;
   - проверяет права пользователя на указанный `channelType`:
     - для `GLOBAL` — достаточно быть авторизованным;
     - для `SQUAD` — пользователь должен состоять в отряде;
     - для `COMPANY` — отряд пользователя должен состоять в роте;
   - формирует нормализованное сообщение с:
     - `channelType` и `channelId` (например, id отряда или роты),
     - данными автора (id, nickname, avatarIcon),
     - текстом и временем отправки `sentAt`;
   - публикует его во внутренний канал (например, Redis `chat:global`, `chat:squad:{id}`, `chat:company:{id}`);
   - ретранслирует сообщение всем подключённым клиентам, подписанным на этот канал.

5. Клиенты всех пользователей, находящихся в данном чате, получают событие:

   ```json
   {
     "type": "CHAT_MESSAGE",
     "channelType": "SQUAD",
     "channelId": 10,
     "messageId": 1,
     "author": {
       "id": 1,
       "nickname": "Viking",
       "avatarIcon": "wolf_01"
     },
     "text": "Двигаемся к точке сбора",
     "sentAt": "2025-01-01T12:00:00Z"
   }
   ```

6. Интерфейс клиента добавляет сообщение в текущую локальную ленту чата.

**Альтернативы/ошибки:**

- Пользователь пытается отправить сообщение в `SQUAD`, не будучи в отряде, или в `COMPANY`, не будучи в роте → сервер возвращает через WebSocket сообщение об ошибке (`type: "ERROR"`), а сообщение не рассылается.
- Невалидный формат сообщения (`action` не `SEND_MESSAGE`, пустой `text` и т.п.) → сообщение об ошибке (`BAD_REQUEST`).

### Особенности отсутствия истории чатов

1. При первоначальном подключении к WebSocket клиент **не** получает историю сообщений ни для одного канала.
2. Клиент видит только те сообщения, которые приходят **после** установления текущего WebSocket‑соединения.
3. При переподключении:
   - предыдущее состояние ленты сообщений на клиенте формируется только из локально сохранённого состояния (если фронт сохраняет);
   - backend не отправляет историю за время отсутствия соединения.
4. При смене состояния пользователя:
   - вступление в отряд → при следующем подключении (или сразу, если логика поддерживает динамическую перенастройку) клиент начинает получать сообщения отрядного чата;
   - выход из отряда → клиент перестаёт получать новые сообщения отрядного чата;
   - аналогично для ротного чата при вступлении/выходе отряда из роты.
