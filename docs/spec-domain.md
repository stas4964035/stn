## Роли и статус аккаунта

### Роли
- `USER`
- `MODERATOR` (права модерации/админ-панели)
- `ADMIN`

### accountStatus
- `ACTIVE`
- `BLOCKED`
- `DELETED`

Правила:
- Если `accountStatus != ACTIVE`:
  - любой защищённый REST endpoint должен отвечать `403` и code `ACCOUNT_BLOCKED` или `ACCOUNT_DELETED`
  - любое WS соединение должно быть отклонено/закрыто с `ERROR` (в формате ErrorResponse)

## JWT

- Алгоритм: HS256
- Claims (MVP): `sub` = userId, `email`, `role`
- TTL токена: 24h

### Инвалидация токенов при изменении accountStatus (канон, conservative)
Требование: “токены недействительны с момента BLOCKED/DELETED”.

Правило реализации MVP:
- Каждый защищённый REST запрос MUST загружать пользователя по `sub` и проверять `accountStatus`.
- Каждое WS соединение MUST загружать пользователя по `sub` и проверять `accountStatus` при подключении.
- Для long-lived WS соединений сервер MUST закрывать сессию, если `accountStatus` меняется на не-ACTIVE.

Это делает ранее выданные токены фактически непригодными после смены статуса без введения blacklist в MVP.

## Инварианты домена (MVP)

### Отряд (Squad)
- Пользователь может состоять максимум в одном отряде.
- У отряда ровно один командир (который также участник).
- Вступление в отряд возможно только если `isOpen = true`.
- Командир может:
  - исключать участников
  - передавать роль командира
  - распускать отряд
- Правила выхода:
  - если командир выходит и участники остаются, система назначает нового командира и эмитит `BECAME_COMMANDER`
  - если выходит последний участник, отряд удаляется.

### Компания (Company)
- Отдельного командира компании нет.
- Управление компанией осуществляется командирами отрядов, входящих в компанию.
- Отряд может принадлежать максимум одной компании.
- Компания может быть открытой или закрытой (`isOpen`).
- Вступление отряда в компанию возможно только если `isOpen = true`.
- Командир любого отряда, входящего в компанию, может:
  - привязывать/отвязывать свой отряд;
  - изменять параметры компании;
  - инициировать роспуск компании, если его отряд единственный в компании.
- Если после отвязки отрядов в компании не осталось ни одного отряда, компания удаляется.

### Метки (Markers)
Метка создаётся на основе `TacticalMarkerType`.

Видимость метки определяется:

- принадлежностью автора к отряду;
- параметрами `TacticalMarkerType`;
- флагом `sendToCompany`, переданным при создании.

Правила:

- Метка всегда видна членам отряда автора.
- Если `TacticalMarkerType.canSendToCompany = true`
  и при создании указан `sendToCompany = true`,
  метка также видна всем членам отрядов,
  входящих в ту же компанию.
- Если `expiresAt` передан в запросе — использовать его;
- Иначе, если указан `defaultLifetimeSeconds`, сервер вычисляет `expiresAt`;
- иначе `expiresAt=null`.
- Listing по умолчанию: `includeExpired=false` (возвращать только активные).

#### Модель хранения (append-only)

Метки хранятся по модели append-only:

- Метка никогда не удаляется физически.
- Деактивация метки осуществляется установкой `expiresAt = now(UTC)`.

REST семантика:
- `DELETE /api/v1/markers/{markerId}` должна выполнять деактивацию через `expiresAt = now(UTC)`, без физического удаления.
- Активной считается метка, если:
  - `expiresAt == null`, или
  - `expiresAt > now(UTC)`.

Фоновый sweep:

- Сервер периодически проверяет метки с истекшим `expiresAt`.
- Для каждой такой метки должен быть эмитирован `MARKER_DELETED`
  (см. spec-ws.md).
- Sweep не удаляет запись физически, а лишь гарантирует
  корректную эмиссию событий.

#### Уникальность меток (uniquenessPolicy)

Если используется `TacticalMarkerType`, он может определять правило уникальности:

`uniquenessPolicy ∈ {NONE, ONE_PER_USER, ONE_PER_SQUAD}`

Поведение:

- `NONE`
  - Ограничений по количеству активных меток данного типа нет.

- `ONE_PER_USER`
  - У одного пользователя может быть не более одной активной метки
    данного типа одновременно.
  - При создании новой метки того же типа предыдущая активная
    метка пользователя должна быть деактивирована
    (сервер устанавливает `expiresAt = now(UTC)`).

- `ONE_PER_SQUAD`
  - В рамках одного отряда может существовать не более одной
    активной метки данного типа.
  - При создании новой метки предыдущая активная метка
    данного типа в этом отряде должна быть деактивирована.

Деактивация метки считается обычным истечением,
и должна приводить к эмиссии события `MARKER_DELETED`
(см. spec-ws.md).

### Приказы (Orders)
- Приказ принадлежит отряду.
- Статус: `CREATED | IN_PROGRESS | COMPLETED`
- При переходе в `COMPLETED` устанавливается `completedAt`.

### Геопозиции (Geo)

Правила видимости

- Пользователь всегда видит позиции членов своего отряда.
- Если отряд пользователя состоит в компании,
  пользователь дополнительно видит позиции
  командиров отрядов, входящих в ту же компанию.


- `POST /geo/position` — append-only история.
- `GET /geo/positions` — возвращает **последнюю** позицию по видимым пользователям.

## Ошибки (единая каноническая модель)

### ErrorResponse
Используется в REST и WS `ERROR`.

```json
{
  "timestamp": "2025-01-01T12:00:00Z",
  "status": 400,
  "code": "VALIDATION_ERROR",
  "message": "Human readable message",
  "path": "/api/v1/...",
  "details": { }
}
```

### Канонические error codes (MVP)

Auth / user:
- `UNAUTHORIZED` (401)
- `INVALID_CREDENTIALS` (401)
- `USER_ALREADY_EXISTS` (409)
- `ACCOUNT_BLOCKED` (403)
- `ACCOUNT_DELETED` (403)

Squad / company:
- `SQUAD_NOT_FOUND` (404)
- `COMPANY_NOT_FOUND` (404)
- `ALREADY_IN_SQUAD` (409)
- `FORBIDDEN` (403)

Markers / orders:
- `MARKER_NOT_FOUND` (404)
- `ORDER_NOT_FOUND` (404)
- `INVALID_ORDER_STATUS` (400)

Generic:
- `VALIDATION_ERROR` (400)
- `CONFLICT` (409)
- `INTERNAL_ERROR` (500)

Примечания:
- `details` зарезервирован под field errors (validation) и контекстные метаданные.
