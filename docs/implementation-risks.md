# implementation-risks.md — риски рассинхронизации реализации с ТЗ

Документ фиксирует риски, при которых реализация или технические спецификации `/docs/spec-*.md` могут уйти от канона `tz_tactical_app.md`.
Цель реестра — заранее выявлять неоднозначности и закрывать их через синхронизацию спецификаций с ТЗ.

## 1) Неоднозначность каноничности REST-путей

- В `spec-api.md` указан base path `/api/v1`, но все endpoint'ы далее записаны без префикса (например, `/auth/register`, `/users/me`, `/squads`).
- Для части команд и WS-trigger'ов в этом же документе и в `spec-ws.md` используются уже полные пути с `/api/v1/...`.

Риск:
- часть команды реализует контроллеры с префиксом `/api/v1`, часть — без него;
- сложнее писать автотесты и клиентские SDK.

Рекомендация:
- явно зафиксировать правило: endpoint в таблицах указаны "relative to base path" или всегда писать полный путь.

## 2) Конфликт модели связи Company↔Squad в текстах

- `spec-infra.md` в ключевых моментах ссылается на `company_squads`.
- Каноническая SQL-миграция (`V1__init_schema.sql`) реализует связь через `squads.company_id`, без join-таблицы `company_squads`.

Риск:
- доменная/репозиторная реализация может расходиться с БД;
- возможны ложные ожидания при проектировании индексов, запросов и каскадов.

Рекомендация:
- синхронизировать формулировку в `spec-infra.md` с фактическим каноном миграции.

## 3) Неполная спецификация переходов статусов приказов

- В `spec-domain.md` есть код ошибки `INVALID_ORDER_STATUS`.
- Явная таблица допустимых переходов (`CREATED -> IN_PROGRESS -> COMPLETED`) не зафиксирована как строгий автомат.

Риск:
- разные сервисы могут по-разному валидировать переходы (например, разрешить `CREATED -> COMPLETED` напрямую).

Рекомендация:
- добавить в `spec-domain.md` формальную таблицу переходов и требования к `completedAt`.

## 4) Не до конца описана семантика уникальности меток

- Для `TacticalMarkerType` введены `uniquenessPolicy = NONE | ONE_PER_USER | ONE_PER_SQUAD`.
- Но не зафиксировано, уникальность проверяется среди всех меток или только активных (не истекших/не деактивированных).

Риск:
- разные SQL-ограничения и бизнес-правила в разных частях системы;
- трудно обеспечить консистентность между REST-валидацией и фоновым sweep.

Рекомендация:
- явно описать, что uniqueness применяется к активному срезу и как учитывается `expiresAt`.

## 5) Не определено поведение sendToCompany при отсутствии company

- Спеки фиксируют, что метка может эмититься дополнительно в канал COMPANY при `sendToCompany = true`.
- Но не определено, что делать, если отряд автора не состоит в компании.

Риск:
- неодинаковое поведение: молча игнорировать флаг, возвращать `400`, возвращать `403`.

Рекомендация:
- зафиксировать единый результат (например, `400 VALIDATION_ERROR` с деталями).

## 6) Нет строгого контракта details для ошибок валидации

- `spec-domain.md` определяет `details` как зарезервированное поле, но не фиксирует конкретную JSON-структуру.

Риск:
- фронт/мобильные клиенты не смогут унифицированно парсить валидационные ошибки.

Рекомендация:
- утвердить каноничный вид `details.errors[]` (например, `{field, message, code}`).

## 7) Не закрыт вопрос авто-подписок WS: MUST или optional

- В `spec-ws.md` раздел "Подписки" помечен как "рекомендуемое поведение MVP".
- При этом для клиентской логики это может быть критичным поведением (получение событий сразу после connect).

Риск:
- серверы разных окружений/веток будут по-разному вести себя при connect;
- нестабильные e2e-тесты realtime.

Рекомендация:
- определить обязательность: либо MUST авто-подписка, либо только client-driven модель.

## 8) Недоопределенность инвалидации long-lived WS-сессий

- `spec-domain.md` требует закрывать WS-сессии, если `accountStatus` меняется на не-ACTIVE.
- Технический механизм не описан (пуш события из auth, polling, recheck на каждое входящее сообщение, TTL session cache).

Риск:
- требование будет реализовано частично или с большими задержками;
- различия поведения между инстансами.

Рекомендация:
- добавить минимальный механизм MVP (например, периодический recheck + верхняя граница задержки).

## 9) Возможная путаница из-за legacy ТЗ

- В `tz_tactical_app.md` есть положения, конфликтующие с spec-каноном (например, автогенерация имен отрядов/рот; вариативная стратегия хранения геопозиции).
- В `spec-domain.md` указан приоритет и более низкий статус ТЗ, но на практике команды часто читают ТЗ отдельно.

Риск:
- реализация по устаревшему/низкоприоритетному документу.

Рекомендация:
- в `tz_tactical_app.md` добавить заметный баннер, что для MVP authoritative только `spec-*`.

## 10) Недостаточная определенность контрактов идемпотентности

- Для команд, которые триггерят важные события (`leave/disband/remove-squad`, деактивация маркера), не описано поведение при повторном запросе/ретраях.

Риск:
- дублирующиеся WS-события, расхождения в состоянии клиента и сервера.

Рекомендация:
- зафиксировать идемпотентные ответы и политику повторной эмиссии событий.
