
# Техническое задание: Тактический навигационный и коммуникационный сервис

## 0. Общая информация

**Проект:**  
Тактический навигационный и коммуникационный сервис для тактических игр (страйкбол и аналогичные).

**Цель:**  
Создать веб-приложение, которое позволяет:

- игрокам регистрироваться и состоять **ровно в одном отряде** одновременно;
- отрядам объединяться в **роты** (объединения отрядов с общей задачей);
- отображать на карте:
  - геопозиции членов своего отряда,
  - геопозиции командиров отрядов, входящих в одну роту,
  - тактические метки от своего отряда и от командиров отрядов роты;
- обмениваться сообщениями в **общем, ротном и отрядном чатах**;
- создавать и отслеживать **приказы** внутри отрядов;
- использовать игровой статус «жив / мертвяк» и аватарки пользователей.

**Технологический стек (целевой):**

- Backend: Java (LTS 17/21), Spring Boot 3.x, PostgreSQL, Redis, WebSocket, JWT.
- Frontend: web-клиент (MVP — простой, без завязки на конкретный фреймворк; возможен переход к SPA).

### Работа со временем и аудит полей `created_at` / `updated_at`

- Все временные значения на backend-е хранятся и обрабатываются в формате UTC.
- В базе данных для ключевых доменных сущностей (пользователи, отряды, роты и т.п.) предусмотрены поля:
  - `created_at` — время создания записи;
  - `updated_at` — время последнего обновления записи.
- На уровне JPA-модели эти поля будут представлены как:
  - `Instant createdAt` → `created_at`;
  - `Instant updatedAt` → `updated_at`;
  и заполняться автоматически средствами Spring Data JPA auditing (`@CreatedDate`, `@LastModifiedDate`).


Требования:

1. **Единый формат времени**
   - В коде backend’а используется тип `Instant` для временных полей, соответствующих `created_at` / `updated_at`.
   - Во внешнем API (REST) значения этих полей при необходимости возвращаются в формате ISO 8601 в UTC (например, `2025-11-28T11:39:00Z`).

2. **Автоматическое заполнение аудиторских полей**
   - Backend включает JPA-аудитинг (`@EnableJpaAuditing`).
   - Поля `created_at` / `updated_at` заполняются и обновляются автоматически на уровне ORM, без явного присваивания в бизнес-логике.

3. **Использование в бизнес-логике**
   - Бизнес-логика может использовать `createdAt` / `updatedAt` для вычисления и отображения времени создания/обновления, но эти поля не являются частью публичного контракта каждого эндпоинта по умолчанию.
   - При необходимости включения этих полей в ответ конкретного REST-метода это явно описывается в спецификации endpoint’a (`backend-api-spec-from-use-cases.md`).

4. **Инфраструктура получения текущего времени (TimeProvider)**
   - В backend вводится интерфейс `TimeProvider` (пакет `com.k44.STN.common.time`), инкапсулирующий получение текущего момента времени в UTC.
   - Реализация по умолчанию `SystemTimeProvider` использует Spring-бин `Clock`, сконфигурированный как `Clock.systemUTC()` в конфигурации `TimeConfig`.
   - Все сервисы и компоненты, которым требуется текущее время в бизнес-логике (расчёт `expiresAt`, проставление временных меток вне JPA-аудита и т.п.), должны получать его через `TimeProvider.now()`, а не через прямые вызовы `Instant.now()` / `LocalDateTime.now()`.
   - В модульных и интеграционных тестах допускается переопределение `TimeProvider` / `Clock` для получения детерминированного времени (fixed clock), чтобы упростить проверку сценариев, завязанных на время.

Данный раздел описывает требования к аудиту и инфраструктуре работы со временем и задаёт единые правила работы со временем в проекте.

## 2. Роли и права доступа

### 2.1. Глобальные роли (User.systemRole)

- **ADMIN**
  - Управление пользователями (назначение/снятие ролей, блокировка/разблокировка).
  - Управление отрядами и ротами (просмотр, роспуск, принудительные изменения).
  - Управление типами тактических меток (`TacticalMarkerType`): добавление, изменение, отключение.
  - Модерация всего контента (метки, приказы, чаты) через админ-интерфейс.

- **MODERATOR**
  - Управление отрядами и ротами (роспуск, удаление отрядов из рот, кик участников).
  - Модерация контента: возможность удалять тактические метки, приказы, чат-сообщения.
  - НЕ управляет глобальными ролями пользователей.
  - НЕ управляет типами тактических меток.

- **USER**
  - Регистрация, вход.
  - Создание отряда (если не состоит ни в одном отряде).
  - Вступление в открытый отряд (если сейчас не состоит ни в одном отряде).
  - Участие в отряде, роте, чатах, метках и приказах в рамках своих прав.

в публичных выдачах данные пользователя могут быть анонимизированы (например, nickname="Deleted user", avatarIcon=null), если этот пользователь фигурирует как автор меток/приказов/чат-сообщений.

### 2.1.1 Статус аккаунта (User.accountStatus)

Помимо глобальной роли (`systemRole`), пользователь имеет отдельный статус аккаунта (`accountStatus`).

`accountStatus` ∈ {`ACTIVE`, `BLOCKED`, `DELETED`}

- `ACTIVE` — пользователь может выполнять разрешённые действия в системе.
- `BLOCKED` —
  - пользователь не может выполнять защищённые действия приложения (REST/WS);
  - вход (/auth/login) запрещён;
  - действующие JWT считаются недействительными с момента блокировки.
- `DELETED` —
  - вход запрещён;
  - действующие JWT считаются недействительными;
  - в публичных выдачах данные пользователя могут быть анонимизированы
    (например, nickname="Deleted user", avatarIcon=null).

`accountStatus` не является ролью и не влияет на набор прав доступа напрямую.
Права определяются `systemRole`, а возможность пользоваться системой —
значением `accountStatus`.
### 2.2. Роли внутри отряда

- **Командир отряда**
  - Определяется полем `Squad.commander`.
  - Всегда входит в `Squad.members`.

  Права:

  - изменять данные отряда (описание, имя, флаг открытости);
  - кикать участников отряда;
  - передавать командование другому участнику отряда;
  - распускать отряд;
  - создавать приказы;
  - менять статус приказов;
  - создавать командирские тактические метки;
  - управлять участием отряда в роте (вступление/выход, изменение параметров роты).

- **Солдат**
  - Любой `User`, находящийся в `Squad.members`, не являющийся командиром.

  Права:

  - просмотр информации об отряде и роте;
  - участие в чатах (общем, ротном, отрядном);
  - создание доступных ему типов тактических меток (например, `ENEMY_SPOTTED`);
  - просмотр приказов отряда;
  - выход из отряда.

## 3. Сущности (модель данных)

### 3.1. User

Поля (логический состав):

- `id`
- `nickname` — позывной/отображаемое имя.
- учетные данные (логин/email, пароль/хэш и т.д.).
- `systemRole` ∈ {`ADMIN`, `MODERATOR`, `USER`}
- `isAlive: boolean`
  - `true` — пользователь находится в игровом состоянии «жив»;
  - `false` — пользователь находится в игровом состоянии «мертвяк».
- `accountStatus` ∈ {`ACTIVE`, `BLOCKED`, `DELETED`}
- `avatarIcon` — идентификатор аватарки.
- `squadId` — опциональный FK на Squad, «текущий отряд пользователя».

Правила:

- при регистрации:
  - `systemRole = USER`,
  - `avatarIcon` назначается случайно из предопределённого набора;
- пользователь может сменить `avatarIcon` на другой из разрешённого списка;
- игровое состояние (`isAlive`):
  - может изменяться самим пользователем (через UI);
  - ADMIN/MODERATOR могут корректировать значение `isAlive` (опционально, в админке).
- в каждый момент времени пользователь может состоять **ровно в одном отряде** (или ни в одном).

### 3.2. Squad (Отряд)

Поля:

- `id`
- `name`
- `description`
- `commander: User`
- `members: Set<User>` (включая `commander`)
- `isOpen: boolean` — открыт/закрыт для вступления.
- `color` — цвет отряда (из палитры «радуга + белый»).
- `company: Company | null`

Правила:

- пользователь может состоять максимум **в одном** отряде:
  - при создании нового отряда пользователь не должен состоять в другом отряде;
  - при вступлении в отряд пользователь не должен состоять в отряде;
- при создании:
  - `name` генерируется автоматически: `Отряд [хищное животное или ядовитая змея]` (может быть изменён командиром);
  - `color` выбирается случайным образом из фиксированной палитры;
  - создатель становится `commander` и добавляется в `members`;
  - `isOpen = true` по умолчанию.

Автоудаление:

- если `members` пуст — отряд удаляется из системы;
- пользователь, покидая отряд, перестаёт иметь связь с ним.

### 3.3. Company (Рота)

Поля:

- `id`
- `name`
- `description`
- `isOpen: boolean`
- `squads: Set<Squad>`

Правила:

- отдельной роли «командир роты» нет; управляют ротой командиры входящих отрядов;
- при создании роты:
  - инициатор — командир отряда;
  - генерируется `name`: `Рота [крупная река или водоём в РФ]` (может быть изменён);
  - задаются `description`, `isOpen`;
  - отряд-инициатор включается в `squads`;
- один отряд может состоять максимум в одной роте;
- если `squads` пуст — рота удаляется.

### 3.4. GeoLocation

Отдельная сущность/таблица.

Поля:

- `id`
- `userId`
- `lat`, `lon`
- `timestamp`

Назначение:

- хранение актуальной геопозиции пользователей;
- стратегия хранения (одна последняя точка или история) выбирается при реализации, но для фронта важна последняя.

Отображение:

- пользователь видит на карте:
  - геопозиции всех членов своего отряда;
  - геопозиции командиров отрядов, входящих в ту же роту.

Источник координат:

- пользователь может выбрать режим:
  - **автоматический** — координаты берутся с устройства (GPS/гео-API браузера) при наличии разрешения и отправляются с заданной периодичностью;
  - **ручной** — пользователь кликает по карте и тем самым задаёт своё положение, которое отправляется на сервер.

### 3.5. TacticalMarkerType

Тип (шаблон) тактической метки, конфигурируемый ADMIN.

Поля:

- `id`
- `key` — системный ключ (`"ENEMY_SPOTTED"`, `"ATTACK"`, `"MOVE"`, `"HOLD"`, `"MISSION_TARGET"`, `"RESPAWN_POINT"`, и т.п.).
- `name` — отображаемое название.
- `defaultDescription`
- `icon`
- `defaultLifetimeSeconds` (может быть `null`).
- `roleRestriction` ∈ {`ANY_MEMBER`, `COMMANDER_ONLY`}
- `canSendToCompany` (bool)
- `uniquenessPolicy` ∈ {`NONE`, `ONE_PER_USER`, `ONE_PER_SQUAD`}
- `category` (опционально)
- `active` (bool)

Инициализация (пример):

- `ENEMY_SPOTTED`:
  - `roleRestriction = ANY_MEMBER`,
  - `defaultLifetimeSeconds = 600` (10 минут),
  - `canSendToCompany = true` (метка видна всему отряду и всем отрядам роты),
  - `uniquenessPolicy = NONE`.
- `ATTACK`, `MOVE`, `HOLD`:
  - `COMMANDER_ONLY`, `canSendToCompany = true`, `uniquenessPolicy = ONE_PER_USER`.
- `MISSION_TARGET`, `RESPAWN_POINT`:
  - `COMMANDER_ONLY`, `canSendToCompany = true`.

Система должна позволять в будущем добавить тип `ORDER`, если потребуется визуализировать приказы на карте.

### 3.6. TacticalMarker

Конкретная метка на карте.

Поля:

- `id`
- `markerType: TacticalMarkerType`
- `author: User`
- `squad: Squad`
- `company: Company | null` — если метка ротная
- `lat`, `lon`
- `description`
- `createdAt`
- `expiresAt` (может быть `null`)

Правила:

- автор должен иметь права согласно `markerType.roleRestriction`;
- если метка ротная, тип должен иметь `canSendToCompany = true`;
- если указан `defaultLifetimeSeconds`, при создании вычисляется `expiresAt`;
- при `uniquenessPolicy = ONE_PER_USER`/`ONE_PER_SQUAD` старые активные метки соответствующего типа удаляются/деактивируются;
- тип `ENEMY_SPOTTED`:
  - доступен всем членам отряда;
  - по умолчанию живёт 10 минут;
  - такая метка отображается:
    - всем членам отряда-инициатора,
    - всем членам других отрядов той же роты (если отряд состоит в роте и пользователь при создании метки поставил флаг "транслировать в роту");
- пользователь видит:
  - все метки своего отряда;
  - ротные метки командиров других отрядов роты (если тип и конкретная метка это разрешают).

### 3.7. Order (Приказ)

Поля:

- `id`
- `squad: Squad`
- `author: User` — командир отряда
- `description` — текст приказа
- `status` ∈ {CREATED, IN_PROGRESS, COMPLETED}
- `createdAt`
- `completedAt` (при `COMPLETED`)

Правила:

- приказ относится к одному отряду;
- создаёт только командир отряда;
- допустимый переход статуса: `CREATED → IN_PROGRESS → COMPLETED` (обратного нет);
- при удалении/расформировании отряда все приказы отряда удаляются вместе с ним;
- пользователям показываются все приказы отряда (активные и выполненные, с пометкой по статусу).

*(На будущее допустима привязка приказа к метке через добавление `markerId`/`orderId`.)*

## 4. Функциональные требования

### 4.1. Регистрация и аутентификация

- Регистрация нового пользователя:
  - `systemRole = USER`,
  - случайный `avatarIcon` из набора.
- Вход:
  - аутентификация по логину/паролю;
  - выдача JWT токена.
- ADMIN может менять `systemRole` пользователя.
- Пользователь может менять своё игровое состояние (`isAlive`) через клиент.
### 4.1.1 Поведение клиента
- При получении ответа “аккаунт заблокирован” клиент должен:
  - разлогинить пользователя;
  - показать экран блокировки (текст может брать из message).

### 4.2. Управление отрядами

**Создание отряда**

- Доступно пользователю, который **не состоит ни в одном отряде**.
- При создании:
  - генерируется `name`,
  - выбирается `color`,
  - создаётся `Squad` с:
    - `commander = creator`,
    - `members = { creator }`,
    - `isOpen = true`;
  - пользователь связывается с этим отрядом (его единственный отряд).

**Вступление в отряд**

- Пользователь может вступить в открытый отряд (`isOpen = true`), если:
  - авторизован,
  - **не состоит в другом отряде**.
- Вступление добавляет пользователя в `members` и делает этот отряд его текущим.

**Выход из отряда**

- Любой участник может выйти из своего отряда.
- Если выходит солдат:
  - удаляется из `members`,
  - если после этого `members` пуст — отряд удаляется.
- Если выходит командир:
  - если в `members` остались другие участники:
    - командование передаётся одному из них (простой алгоритм: первый в списке/первый по id);
    - командир удаляется из `members`, перестаёт состоять в отряде;
  - если других участников нет:
    - отряд удаляется.

**Открытость отряда**

- Командир может переключать `isOpen`.
- MODERATOR/ADMIN могут менять `isOpen` любого отряда.

**Кик участника**

- Командир может кикнуть любого участника (кроме себя) из своего отряда.
- MODERATOR/ADMIN могут кикнуть любого участника из любого отряда.
- После кика:
  - пользователь удаляется из `members` и перестаёт состоять в отряде;
  - если `members` пуст — отряд удаляется.

**Передача командования**

- Командир может назначить новым командиром любого участника отряда.
- После передачи:
  - старый командир остаётся в `members` как солдат (может затем выйти).

**Роспуск отряда**

- Командир может распустить отряд:
  - отряд и все связанные приказы удаляются;
  - пользователи перестают состоять в отряде.
- MODERATOR/ADMIN могут распустить любой отряд.

### 4.3. Управление ротами

**Создание роты**

- Командир отряда может создать роту:
  - генерируется `name`,
  - задаются `description`, `isOpen`,
  - его отряд добавляется в `squads`.

**Вступление/выход отряда**

- Командир может:
  - вступить с отрядом в открытую роту (`isOpen = true`), если отряд сейчас не состоит в роте;
  - вывести свой отряд из роты.
- Если после выхода всех отрядов `squads` пуст — рота удаляется.

**Управление параметрами роты**

- Любой командир отряда, входящего в роту, может:
  - изменить `name`,
  - изменить `description`,
  - изменить `isOpen`.
- MODERATOR/ADMIN могут удалять/распускать роты и удалять отряды из рот.

### 4.4. Геолокация и карта

- Клиент может работать в двух режимах задания координат пользователя:
  - **автоматический режим**:
    - приложение запрашивает доступ к геолокации устройства;
    - при наличии разрешения с заданной периодичностью отправляет актуальные `lat/lon` на backend;
  - **ручной режим**:
    - пользователь вручную указывает своё положение на карте (клик или drag метки);
    - при изменении положения координаты отправляются на backend.

- Backend сохраняет историю точек в `GeoLocation`.
- На карте пользователь видит:
  - последнюю записанную позицию всех членов своего отряда,
  - последнюю записанную позицию командиров отрядов, входящих в ту же роту.
- При отображении используется:
  - `avatarIcon` пользователя,
  - `color` отряда.

### 4.5. Тактические метки

**Создание меток участниками**

- Пользователь может создавать метки типов, у которых `roleRestriction = ANY_MEMBER`.
- Тип `ENEMY_SPOTTED`:
  - доступен всем членам отряда;
  - по умолчанию имеет срок жизни 10 минут;
  - при создании такой метки:
    - она отображается всем членам отряда-инициатора;
    - если отряд входит в роту, метка также отображается всем членам других отрядов этой роты (ротный уровень видимости).

**Создание меток командирами**

- Командир может создавать метки типов с `roleRestriction = COMMANDER_ONLY`.
- Если `markerType.canSendToCompany = true`, при создании доступен флаг «отправить в роту»:
  - при установке флага метка отображается у всех членов отрядов роты.
- Для типов с `uniquenessPolicy` (например, `ATTACK`, `MOVE`, `HOLD`):
  - при создании метки такого типа удаляются/деактивируются существующие активные метки этого типа от этого пользователя (или отряда — по настройке).

**Управление метками**

- Автор метки может удалить свои метки.
- MODERATOR/ADMIN могут удалять любые метки.

### 4.6. Приказы

**Создание приказа**

- Только командир отряда.
- Командир задаёт текст `description`.
- Создаётся `Order`:
  - `status = CREATED`,
  - `createdAt = now`.
- Приказ отображается всем членам отряда в списке приказов.

**Просмотр приказов**

- Любой член отряда:
  - может получить список активных приказов;
  - может получить список всех приказов отряда (активных и выполненных).
- При вступлении в отряд клиент запрашивает список приказов отряда.

**Изменение статуса приказа**

- Менять статус может:
  - командир данного отряда,
  - ADMIN/MODERATOR (при необходимости).
- Разрешён переход:
  - `CREATED → IN_PROGRESS → COMPLETED`, при этом заполняется `completedAt`.
- Обратный переход `COMPLETED → ACTIVE` не допускается.

**Удаление приказов**

- В обычной работе приказы не удаляются пользователями.
- При удалении/распуске отряда:
  - все связанные приказы удаляются.
- В UI выполненные приказы отображаются с пометкой «выполнен».

**Ограничения**

- Лимита на количество активных приказов в одном отряде нет.

### 4.7. Чаты (WebSocket)— future feature(НЕ ВХОДИТ В MVP)

Три логических чата:

1. **Глобальный чат**
   - доступен всем авторизованным пользователям;
   - канал: `chat:global`.

2. **Чат отряда**
   - доступен участникам `Squad.members`;
   - канал: `chat:squad:{squadId}`.

3. **Чат роты**
   - доступен участникам отрядов, входящих в `Company`;
   - канал: `chat:company:{companyId}`.

Технические требования:

- Общение через WebSocket между клиентом и backend.
- Backend может использовать Redis Pub/Sub для доставки сообщений между инстансами.
- Сообщения **не сохраняются** в PostgreSQL и не сохраняются как история в Redis:
  - нет истории чата;
  - пользователь видит только сообщения, поступившие после его подключения.
- При роспуске отряда/роты:
  - backend прекращает использовать соответствующие каналы.

Формат сообщений (логически):

- `channelType` ∈ {`GLOBAL`, `SQUAD`, `COMPANY`}
- `channelId` (для `SQUAD`/`COMPANY`)
- `authorId`, `authorNickname`
- `text`
- `sentAt`

### 4.8. Уведомления и события

**Минимальный набор событий:**

- Уровень пользователя/отряда:
  - `SQUAD_CREATED`
  - `BECAME_COMMANDER`
  - `JOINED_SQUAD`
  - `LEFT_SQUAD`
  - `KICKED_FROM_SQUAD`
  - `SQUAD_DISBANDED`
- Уровень роты:
  - `SQUAD_JOINED_COMPANY`
  - `SQUAD_LEFT_COMPANY`
  - `COMPANY_DISBANDED`
- Метки:
  - `MARKER_CREATED`
  - `MARKER_DELETED` (включая автоматическое истечение)
- Приказы:
  - `ORDER_CREATED`
  - `ORDER_STATUS_CHANGED`

**Механизм реализации:**

- Действия домена порождают доменные события.
- Для доставки на клиент:
  - каждое событие сериализуется в DTO, например:

    ```json
    {
  "type": "EVENT",
  "eventType": "ORDER_CREATED",
  "channel": "SQUAD",
  "channelId": 10,
  "payload": { "orderId": 700, "squadId": 10 }
}
}
    ```

  - и отправляется:
    - напрямую по WebSocket,
    - либо через Redis Pub/Sub (например, канал `events:squad:{squadId}`), откуда WebSocket-слой доставляет его клиентам.

- Каналы подписки клиентов:
  - персональный канал пользователя (например, `user:{userId}`) для персональных событий;
  - `events:squad:{squadId}` — для событий по отряду;
  - `events:company:{companyId}` — для событий по роте.

Хранение уведомлений:

- В MVP уведомления не сохраняются в БД:
  - доставка только в режиме real-time;
  - нет истории уведомлений.

Функциональность чатов (GLOBAL, SQUAD, COMPANY), описанная ранее,
не реализуется в текущей версии приложения (MVP).

Причины:
- основной фокус MVP — управление отрядами, компаниями,
  приказами, метками и геопозициями;
- чат требует отдельного канонического WebSocket-контракта,
  механизма масштабирования и уточнения правил модерации.

В рамках текущей реализации:
- чат отсутствует;
- отдельный WebSocket endpoint для чата не создаётся;
- Redis Pub/Sub для чата не используется.

Возможная реализация чата рассматривается как future feature.

## 5. Нефункциональные требования

### 5.1. Архитектура

- Монолитное Spring Boot приложение с модульной структурой по доменам:
  - `auth`, `users`, `squads`, `companies`, `geo`, `markers`, `orders`, `chat`, `notifications`.
- Чёткое разделение доменной логики и инфраструктуры.

### 5.2. Стек

- Java LTS (17/21), Spring Boot 3.x.
- PostgreSQL для доменных данных.
- Redis:
  - как Pub/Sub для чатов и событий,
  - опционально как кэш.
- WebSocket (возможен STOMP поверх WS).
- JWT для аутентификации.

### 5.3. Безопасность

- Все защищённые операции доступны только при наличии валидного JWT.
- Авторизация по:
  - глобальной роли (`systemRole`),
  - принадлежности к отряду/роте,
  - роли в отряде (командир/солдат) для операций с приказами и командирскими метками.

### 5.4. Тестирование

- Юнит-тесты доменной логики:
  - правила вступления/выхода/кика;
  - автоудаление отрядов/рот;
  - создание/жизненный цикл меток;
  - создание/завершение приказов.
- Интеграционные тесты REST API.
- При возможности — тесты WebSocket-взаимодействия (создание приказов, меток, уведомления).

### 5.5. Ограничения

- Нет истории чата и уведомлений (только real-time).
- Пользователь может состоять только в одном отряде.
- Отряды и роты без участников удаляются без сохранения истории.
- Типы меток управляются через БД, а не через enum’ы в коде.
